## Why now

Maintainer dogfooded the just-published 1.0.6 and saw on every OpenAI cell:

```
‚úó Q1/gpt/5-search   OpenAI: Unrecognized request argument supplied: reasoning_effort
```

Root cause: `SUPPORTS_REASONING_EFFORT` in `lib/providers/openai.js:33-35` whitelists ALL `gpt-5*` models, including `gpt-5-search-api`. Search-variant models are stripped down ‚Äî they don't accept `reasoning_effort`. The current whitelist sends it anyway, OpenAI rejects with HTTP 400, every OpenAI cell fails.

**Severity**: P0. Real-installed 1.0.6 cannot complete a single `run` on OpenAI without operator manually swapping models. Breaks happy path on the most commonly configured provider.

Worse: `test/openai.test.js:43-48` (a test added in the 1.0.2 provider resilience layer) **asserts the buggy behaviour as correct** ‚Äî i.e. expects `gpt-5-search-api + reasoning_effort=high ‚Üí reasoning_effort in body`. The test passes today because it codifies the bug. Without this hotfix the test stays green, the bug stays in the code, and no automated guard catches future regressions of the same shape.

Plus a P1 UX issue: live-rows shows cryptic statuses (`firing‚Ä¶`, `60s pacing`) with no explanation of what's happening, no countdown updates while waiting, and the spinner appears to freeze during API in-flight. Operator can't tell if the process is working or hung.

Closes:
- `[bug ¬∑ P0]` `gpt-5-search-api` (and other search-variants) sent unsupported `reasoning_effort` ‚Üí every OpenAI cell fails
- `[test ¬∑ P0]` `test/openai.test.js:43-48` codifies the bug ‚Äî must be rewritten to assert the correct behaviour
- `[ux ¬∑ P1]` live-rows status labels lack clarity ‚Äî `firing`, `pacing` are jargon; no countdown; no abort hint

## Fix design

### Fix 1 ‚Äî `SUPPORTS_REASONING_EFFORT` regex tightening

**File**: `lib/providers/openai.js:33-35`

**Current** (buggy):
```js
const SUPPORTS_REASONING_EFFORT = (id) =>
  (/^gpt-/i.test(id) && !/^gpt-[0-4](?:\D|$)/i.test(id))
  || /^o[1-9]\d?(?:[-_]|$)/i.test(id);
```

**Target**:
```js
// Reasoning-effort gate. Sends reasoning_effort ONLY to models that accept it.
//
// Whitelist: `gpt-` prefix except gen 0-4 (legacy non-reasoning), plus o-series.
// HARD EXCLUDE: any model ID containing 'search'. Search-variants are
// stripped-down RAG-tuned models that DON'T accept reasoning_effort:
//   - gpt-5-search-api          (the bug that motivated 1.0.7)
//   - gpt-5-mini-search-api
//   - gpt-4o-search-preview     (already worked because excluded by gen-0-4 rule)
//   - hypothetical gpt-6-search, o4-search, etc.
// Sending reasoning_effort to a search-variant returns HTTP 400:
//   "Unrecognized request argument supplied: reasoning_effort"
const SUPPORTS_REASONING_EFFORT = (id) =>
  !/search/i.test(id) && (
    (/^gpt-/i.test(id) && !/^gpt-[0-4](?:\D|$)/i.test(id))
    || /^o[1-9]\d?(?:[-_]|$)/i.test(id)
  );
```

**Why this fix and not another**:
- Rejected: model-by-model allowlist (e.g. explicit `['gpt-5', 'gpt-5-mini', 'o1', ‚Ä¶]`) ‚Äî too brittle, requires code change for every new model OpenAI ships. Regex with future-proof exclusion stays correct for `gpt-6`, `gpt-7`, etc.
- Rejected: catch HTTP 400 at runtime and retry without `reasoning_effort` ‚Äî adds latency, costs an extra API call per init/run, doesn't help the user understand what happened. Upstream gate is cheap and clear.

**Regression test**: Fix 2 below ‚Äî explicit test case that `gpt-5-search-api + reasoning_effort=high` produces a body WITHOUT the field.

### Fix 4 ‚Äî Countdown updates while waiting

**File**: `lib/util/live-rows.js`

When a row has `status: 'cooldown' | 'ledger-wait'` and a `deadlineMs` field, the renderer (already runs every 100ms per `INTERVAL_MS`) should recompute the remaining-seconds string each tick. Currently `detail` is static after the initial `update()` call.

**REV 2 architect catch**: `live-rows.js::update()` (lines 180-186) ONLY forwards `status` and `detail` ‚Äî arbitrary fields are silently dropped. So the original plan's `live.update(taskId, { ..., deadlineMs: Date.now() + ms })` would silently lose `deadlineMs`. Fix 4 must therefore:

1. **Extend `update()` signature + JSDoc typedef** (`lib/util/live-rows.js:36-50, 180-186`) to also forward `deadlineMs`. On transition to `running` / `done` / `error`, explicitly clear it (`row.deadlineMs = null`) so a stale value doesn't keep ticking.

2. **Add `labelPrefix` field** instead of regex prefix-surgery. The status event handler at `bin/aeo-tracker.js:1965-1977` sets both the initial `detail` AND a `labelPrefix` (e.g. `"TPM rate-limit ‚Äî "` and `"provider cooldown ‚Äî "`), then the renderer composes the final string from a template.

**Target ‚Äî `update()` signature**:
```js
function update(taskId, patch) {
  const row = rows.find(r => r.taskId === taskId);
  if (!row) return;
  if ('status' in patch) row.status = patch.status;
  if ('detail' in patch) row.detail = patch.detail;
  if ('deadlineMs' in patch) row.deadlineMs = patch.deadlineMs;
  if ('labelPrefix' in patch) row.labelPrefix = patch.labelPrefix;
  // Clear stale countdown when status moves out of waiting states.
  if (patch.status === 'running' || patch.status === 'done' || patch.status === 'error') {
    row.deadlineMs = null;
    row.labelPrefix = null;
  }
}
```

**Target ‚Äî renderer countdown**:
```js
function rerender(row) {
  if ((row.status === 'cooldown' || row.status === 'ledger-wait') && row.deadlineMs && row.labelPrefix) {
    const remainingMs = Math.max(0, row.deadlineMs - Date.now());
    const sec = Math.ceil(remainingMs / 1000);
    row.displayDetail = sec > 0
      ? `${row.labelPrefix}${sec}s remaining`
      : `${row.labelPrefix}releasing‚Ä¶`;
  } else {
    row.displayDetail = row.detail;
  }
}
```

Renderer (`renderAll()`) calls `rerender(row)` before formatting each row, then uses `row.displayDetail` instead of `row.detail`.

**Status-event handler at `bin/aeo-tracker.js:1965-1977`** (the actual writer):

## Risk + rollback

### Risks

1. **Fix 1 regex over-excludes** ‚Äî what if OpenAI ships `gpt-5-search-pro-reasoning` (hypothetical) that DOES support reasoning_effort? Regex would exclude it. Mitigation: future-proofing is best-effort; if such a model lands, add explicit override via env var or config flag in a follow-up.
2. **Fix 4 countdown can drift** ‚Äî if the renderer doesn't run on time (process busy), the countdown could lag the real wait. Mitigation: countdown is computed from `deadlineMs` (absolute timestamp), not decremented per tick. Catches up automatically next render.
3. **Fix 5 abort hint may look noisy** in CI/non-TTY mode. Mitigation: gated on `isTTY()` ‚Äî same gate live-rows already uses for spinner suppression.

### Rollback

If 1.0.7 introduces a regression: publish `1.0.8` reverting the search-variant exclusion. The bug from 1.0.6 stays but is a known UX issue with documented workaround (`--openai-model=gpt-5` to switch to non-search model).
