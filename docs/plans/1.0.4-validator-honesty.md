# 1.0.4 — Validator honesty hotfix

**Status**: REV 4 — addressing IT-architect's REV 3 review which found 4 blockers via the scenario-wide audit (A1: wrong `validationCache` scope, A2: wrong `opts.geography` name, A3: skip in single-provider path leaves contract violated for common case, A4: temporal contradiction — `formatSelection` renders BEFORE the new validation pass), plus B1 (cmdInitQueries path uncovered) and B2 (test 2 missing rendering assertion).
**Target version**: `1.0.4`
**Target publish**: same day as approval (hotfix for live 1.0.3 issue)

---

## Why now

Maintainer dogfooded 1.0.3 on `typelessform.com` immediately after publish. Result: **the validator's own suggested fix commands get rejected by the validator on the next run**. Three independent root causes surfaced in one session:

1. The `(validated)` tag in the Alternatives pool means "passed industry-fit only" — when the operator takes those queries via `--keywords`, the commercial-only stage re-blocks them. **Contract violation: the tool promises validation, then rejects what it promised.**
2. When the validated pool is exhausted, recovery falls back to filler templates of the form `best ${brand} alternatives 2026` / `top ${brand} competitors` / `${brand} vs alternatives`. For new brands (the exact users who need AEO tracking), the LLM autocorrects the unknown brand name (`typelessform → Typeform`) and answers about an entirely different category — guaranteed blocked by the commercial-only check.
3. For new brands in `--yes` mode, there is currently **no safe path to a saved config**: `--auto` blocks, validated `--keywords` re-block, filler `--keywords` re-block, `--manual` doesn't work in `--yes` (requires interactive). Only `--force` works, and it is labelled "degrades trend data" — operators won't pick it.

Architectural reasoning (now saved as `memory/project_query_archetypes_category_based.md`): AEO visibility is measured by **categorical queries** ("best X for Y") the way real users search, not by **brand-comparison queries** ("X vs alternatives"). The latter only works for brands the LLM already knows — and those brands don't need AEO tracking.

Closes the following items added to `docs/roadmap.md` today (these were the immediate post-1.0.3 dogfood findings — not yet a formal roadmap section, but tracked in this plan):

- `[bug · P0]` `(validated)` tag means only one validator passed → contract violation
- `[bug · P0]` Recovery filler templates use brand-archetype → broken for new brands
- `[bug · architectural]` No safe `--yes` path for new brands → need explicit drop-`--yes` recovery option

Defers all the 1.0.4 items that were in `docs/roadmap.md` before this session (UX redesign, auto-substitute, CLI hygiene, placeholder-domain hint, README `--keywords` mention) to **1.0.5**. This is a focused trust-restoration hotfix, not a feature release.

## Fix A — `(validated)` pool genuinely validated at init time

**REV 3 — Architect's REV 2 implementation audit caught that REV 2's Fix A was STILL dead code in production.** The original plan filtered pool entries by `c.search_behavior` inside `formatRecoveryPanel`, REV 2 attempted to enrich pool entries from `v.updatedCache` at the panel call site. **But `v.updatedCache` only contains verdicts for the 3 currently-config'd queries — pool entries (5 alternatives) are NEVER in that cache.** The join produces zero matches in production; the `!c.search_behavior` fallback branch passes everything through; the filter is structurally a no-op.

**Root cause (architect-traced)**: `search_behavior` is produced ONLY by `validateQueriesWithLLM` (called from `runTwoStageValidation`). The research pipeline's Phase 4 (`runValidation` in `research/research.js:117`) uses **category-validation** (a different function that sets `c.validation === 'ok'` only), NOT the LLM industry-fit check. So pool entries pass through research carrying `validation: 'ok'` but **never receive `search_behavior`**. The data simply does not exist anywhere by the time `cmdInit` builds `config_candidatePool`.

**Real fix**: at init time, after `selectTopThree` produces the 5 alternatives, run them through `runTwoStageValidation` (industry-fit + commercial-only) explicitly. Attach `search_behavior` (plus `valid`, `confidence`) to each pool entry. Persist enriched pool entries to `.aeo-tracker.json`. Filter at panel-boundary now operates on real data.

**Cost analysis**:
- 5 pool entries × 1 LLM call each (industry-fit check is the LLM stage producing `search_behavior`)
- Pool queries are typically NEW to the cache (the cache from research stage contains only the queries selected as final, not the alternatives) → 5 cache misses
- ~$0.005-0.010 per init at current OpenAI gpt-5-mini pricing
- One-time cost at init only — not re-paid on `run`/`report`/`diff`
- Acceptable tradeoff for honest validation: the architect's REV 2 finding (the contract was unenforceable) is unacceptable for a "validator-honesty hotfix" by name.

**Files modified**:
- `bin/aeo-tracker.js::cmdInit` — at the point where `config_candidatePool` is built (currently line 1284-1287), add a pre-persist validation pass on the pool
- `lib/init/validator-recovery.js:202-237` (`formatRecoveryPanel`) — Fix A.3 filter unchanged from REV 2; just operates on already-enriched pool

**Data shape after fix**:
- `candidatePool` entries: `{ text, intent, score, unverified, search_behavior, validVerdict, confidence }`
- The `(validated)` tag rendering at `select.js:170-172` should ALSO be updated to require `c.search_behavior === RETRIEVAL` in addition to `validation === 'ok'` — making the user-facing tag honest at selection time, not just at recovery time.

**Target — Part 1 (REV 4)**: validate the pool **BETWEEN `selectTopThree` and `formatSelection`** in `bin/aeo-tracker.js::cmdInit` — i.e. between current lines 1253 and 1259. This is the only ordering that makes Fix A.1c (`(validated)` label in `formatSelection`) honest, because the selection display reads `alt.search_behavior` directly off the alternative objects.

```js
// === Insert BETWEEN line 1253 (selectTopThree) and line 1259 (formatSelection) ===
//
// 1.0.4 Fix A — Validate pool entries through both validator stages
// (industry-fit + commercial-only) at init time, mutate verdicts onto
// alternative objects so formatSelection sees them, then proceed.
// Without this, pool entries carry only {text, intent, score, unverified}
// from research (which used category-validation, not industry-fit), so the
// recovery panel and the user-facing (validated) tag cannot honestly
// reflect commercial-only status. ~5 LLM calls per init; cache hits
// honoured naturally if any alternatives happen to match existing cache.
const primaryForPoolValidation = researchProviders[0];
const poolAlts = selectResult.alternatives.slice(0, 5);
if (poolAlts.length > 0 && primaryForPoolValidation?.providerCall) {
  try {
    const { runTwoStageValidation } = await import('../lib/init/research/run-validation.js');
    const poolValidation = await runTwoStageValidation({
      queries: poolAlts.map(a => a.text),
      brand, domain, category: categoryDescription,
      geography: geoTags || [],
      primary: primaryForPoolValidation,
      secondary: null,             // single-provider check produces search_behavior fine
      validationCache: [],          // no prior cache in cmdInit (fresh init flow)
      commercialOnly: false,        // we want verdicts for ALL alts, not just informational filtering
    });
    const verdicts = poolValidation.updatedCache || [];
    // Mutate verdicts onto the alternative objects so formatSelection
    // (line 1259) sees them — without this, the (validated) label at
    // select.js:170-172 cannot honestly check both stages.
    for (const a of poolAlts) {
      const verdict = verdicts.find(v => v.query === a.text);
      if (verdict) {
        a.search_behavior = verdict.search_behavior;
        a.confidence = verdict.confidence;
        // Note: a.unverified already reflects category-validation; we keep
        // it as-is. The (validated) label combines both signals.
      }
    }
  } catch (err) {
    // Pool validation failure is non-fatal — fall back to legacy pool
    // (no search_behavior, panel filter pass-through, (validated) label
    // governed by `unverified` only). Yellow warning so the operator knows.
    console.error(`${c.yellow}  Pool validation skipped: ${err.message}${c.reset}`);
  }
}
```

**Target fillers logic**: when `category` is **clean** (short noun phrase ≤4 words), build category-based fillers. Otherwise — including the `--keywords` mode case where category is empty, AND the typical `--auto` case where `inferCategory()` produced a 60-160-char marketing string — drop fillers entirely.

**Architect R2 caught**: `inferCategory()` (`lib/init/fetch-site.js:115-131`) returns title + meta + h1 stripped of brand — a marketing sentence, not a noun phrase. Plugging that into `best ${category} for small business 2026` yields invalid English like `best AEO services for B2B SaaS startups · Visibility tracker for AI search for small business 2026`. Also the hardcoded "small business" / "enterprise" vertical produces nonsense for verticals where small business isn't the audience (e.g. `category="enterprise cloud security"`). REV 2 fix below removes both problems.

```js
// Category-based fillers, domain-agnostic. These mirror how real users
// actually search («best X 2026» / «top X tools»), not brand-comparison
// archetypes which only work for established brands the LLM already knows.
// See memory/project_query_archetypes_category_based.md.
//
// Two guards:
//   (a) category must be non-empty
//   (b) category must look like a noun phrase (≤4 words) — `inferCategory()`
//       often returns a long marketing sentence, in which case interpolation
//       yields invalid English. When that happens we treat category as absent.
//
// No hardcoded vertical clause: `best ${category} 2026` works for any
// category; `best ${category} for small business` would break for verticals
// where small business isn't the audience.
const categoryClean = category && category.split(/\s+/).length <= 4 ? category : '';
const categoryFillers = categoryClean
  ? [
      `best ${categoryClean} 2026`,
      `top ${categoryClean} tools`,
      `${categoryClean} platforms`,
    ]
  : [];

const finalQueries = suggested.length === 3
  ? suggested
  : [...suggested, ...categoryFillers.slice(suggested.length)].slice(0, 3);

// Honest poolNote per state: fully validated / partial+category-templates /
// pool-exhausted-no-category. No more "edit as needed" euphemisms.
const poolNote = suggested.length === 3
  ? ' (from validated pool — both validator stages passed)'
  : suggested.length > 0
    ? categoryClean
      ? ` (${suggested.length} from validated pool, rest are category templates — edit if you have a specific vertical)`
      : ` (${suggested.length} from validated pool — pool exhausted; see options 2-4 below to get more)`
    : categoryClean
      ? ' (category templates — edit if you have a specific vertical in mind)'
      : ' (no validated alternatives, no clean category available — see options 2-4 below for what to do instead)';

// When categoryClean is empty AND no validated suggestions, option 1's
// command would emit no useful keywords. Skip option 1 entirely in that
// case — the operator's next move is options 2/3/4.
const showOption1 = suggested.length > 0 || categoryClean;
```

## Risk + rollback

### Risks

1. **Fix A pool-filter is too aggressive.** If most pool entries lack `search_behavior` (older entries from previous validation passes that wrote partial cache), the filter passes them all through (`!c.search_behavior` is `true`) — which is intentional, but means the contract weakens to "passed industry-fit OR no LLM data available". Mitigation: regression test #2 covers this; document the behaviour in the panel comment.

2. **Fix B category-fillers might still be wrong shape.** Users in obscure verticals (e.g. `category = "AI-driven HVAC predictive maintenance"`) get `best AI-driven HVAC predictive maintenance for small business 2026`, which is awkward English. Acceptable for hotfix — the better fix is LLM-generated category fillers, but that adds an API call and belongs in 1.0.5.

3. **Fix C tells operators to drop `--yes`.** Some CI environments cannot run interactively (no TTY). The panel doesn't yet detect that. Mitigation: option 4 explicitly says "you'll be prompted" so a CI operator knows it won't work in their context. Better detection is a 1.0.5 item.

### Rollback plan

**Primary**: publish `1.0.5` with the revert. Hotfix iteration is the same as 1.0.3 → 1.0.4 we are running now.

**Best-effort fallback**: `npm unpublish aeo-platform@1.0.4` within 72 hours, only if npm policy allows (likely will, since 1.0.4 is hours-old at that point and probably has no dependents).

**Safety net**: `npm install aeo-platform@1.0.3` continues to resolve for anyone pinned. 1.0.3 still has the bugs being fixed here, but it does not crash or corrupt data — it merely traps users in an init loop. Acceptable temporary state if 1.0.4 has a worse regression.
