# 1.0.8 — Validator honesty (часть 2): доверяем `valid:true`, синхронизируем правила

**Статус**: DRAFT — ждёт architect review

## Простыми словами — зачем этот релиз

В 1.0.7 на реальном прогоне всплыли три противоречия, которые читаются как баги в одной картинке:

```
Cannot auto-recover — only 5 of 5 commercial candidates passed validation.
Blocked:
  ✗ "top voice form automation services for ecommerce"
    non-commercial (search_behavior: retrieval-triggered)
```

- Заголовок говорит «5 из 5 прошли» — но recovery panel всё равно появляется
- Подпись говорит «non-commercial» — но `retrieval-triggered` это и есть **commercial OK**
- Настоящая причина блока — LLM сказала «уверена на 65%», а порог жёсткий 70%

Это **тот же класс бага**, который мы лечили в 1.0.4 → 1.0.5 → 1.0.6 → 1.0.7: substitution-блок (silent swap top-3 ↔ spare) использует одно правило, а главная проверка — два правила. Из-за этого top-3 проходят substitution, потом блокируются главной.

Решение: **доверяем `valid:true` целиком**. Если LLM явно сказала «да, валидный запрос» — мы не пересматриваем её решение через дополнительные пороги уверенности. Уверенность 0.6-0.7 для нормальных коммерческих запросов — это **типично** (LLM учитывает альтернативные интерпретации), не повод блокировать.

## 4 конкретных фикса

### Фикс 1. Доверяем `valid:true` (убираем порог уверенности из блокировки)

**Файл**: `lib/init/research/run-validation.js:186`

**Сейчас**:
```js
const llmIssues = llmResults.filter(r => !r.valid || r.confidence < CONFIDENCE_THRESHOLD);
```
Блокирует «`!valid` ИЛИ confidence < 0.7». Запрос с `valid:true, confidence:0.65` блокируется.

**Будет**:
```js
const llmIssues = llmResults.filter(r => !r.valid);
```
Блокирует только когда LLM явно сказала «невалидный». Confidence остаётся в кэше для аудита, но решения не принимает.

**Почему именно так**:
- `valid` — это **явный вердикт** LLM, односимвольный YES/NO
- `confidence` — это **процент её уверенности** в собственном вердикте
- Если LLM сказала YES — нет смысла спрашивать «а ты уверена?» и считать «недостаточно уверена → значит NO»
- В реальности confidence 0.60-0.65 — норма для коммерческих запросов (LLM видит alt-meanings и снижает score). Жёсткий 0.7 отвергает корректные запросы.

`CONFIDENCE_THRESHOLD` остаётся определён в `validate-query-llm.js:26` (на случай если он используется ещё где-то для информационных целей — например в logging). Но в блокирующей логике больше не применяется.

**Тесты для обновления**: `test/validate-queries.test.js:160` — assertion про confidence threshold, нужно переписать что блокировка теперь только по `valid:false`.

### Фикс 2. Substitution использует **те же правила** что главная проверка

**Файлы**:
- `bin/aeo-tracker.js:1362-1368` — main path
- `bin/aeo-tracker.js:817-823` — queries-only path

**Сейчас** (1.0.7 в main):
```js
const PASS = (c) =>
  c.search_behavior === SEARCH_BEHAVIORS.RETRIEVAL
  || (!verdicts.find(v => v.query === c.text));
```
Проверяет только commercial-only. Если LLM сказала `valid:false` — substitution всё равно пропустит query (потому что smотрит только на `search_behavior`).

**Будет**:
```js
const PASS = (c) => {
  const verdict = verdicts.find(v => v.query === c.text);
  // Если нет вердикта — graceful pass (legacy для cache miss / no-provider)
  if (!verdict) return true;
  // Доверяем `valid` field LLM. Дополнительно требуем commercial behavior.
  return verdict.valid === true
      && verdict.search_behavior === SEARCH_BEHAVIORS.RETRIEVAL;
};
```

После Фикса 1 главная проверка блокирует только `!valid`. Substitution теперь требует `valid:true` AND `search_behavior:'retrieval-triggered'`. Те же критерии. Запрос не может пройти substitution и при этом быть заблокирован главной проверкой.

То же изменение симметрично применяется к queries-only branch на :817-823.

### Фикс 3. Подпись блока показывает **настоящую причину**

**Файл**: `lib/init/validator-recovery.js:246-252`

**Сейчас**:
```js
const reason = b.search_behavior
  ? `non-commercial (search_behavior: ${b.search_behavior})`
  : b.message || b.reason || 'low-confidence LLM verdict';
```
Если `search_behavior` field есть — пишет «non-commercial». Но `search_behavior:'retrieval-triggered'` это commercial. Подпись лжёт.

**Будет**:
```js
let reason;
if (b.valid === false) {
  // Main reason: LLM судит запрос недействительным (по любой причине LLM решил)
  reason = b.reason
    ? `LLM rejected: ${b.reason}`
    : 'LLM rejected as invalid';
} else if (b.search_behavior && b.search_behavior !== 'retrieval-triggered') {
  // Commercial-only block: запрос commercial по форме, но AI не вернёт vendor list
  reason = `non-commercial (search_behavior: ${b.search_behavior})`;
} else if (b.message) {
  // Static-validation issue (acronym, malformed)
  reason = b.message;
} else {
  reason = b.reason || 'unspecified block reason';
}
```

Теперь подпись честная:
- «LLM rejected: ...» если LLM сказала `valid:false`
- «non-commercial» только когда `search_behavior` действительно не commercial
- Static issues со своим текстом
- Никаких противоречивых строк типа «non-commercial (retrieval-triggered)»

### Фикс 4. Заголовок учитывает **обе проверки**

**Файл**: `lib/init/validator-recovery.js:235-238`

**Сейчас**:
```js
const headerSummary = typeof commercialPassingCount === 'number'
  ? `only ${commercialPassingCount} of 5 commercial candidates passed validation`
  : `${dedupedBlockers.length} query/queries blocked by validator`;
```
«5 of 5 commercial passed» считает только commercial-only. Не отражает llmIssues.

**Будет** — посчитать ОБА числа, показать честно:
```js
const totalBlocked = dedupedBlockers.length;
const commercialOk = typeof commercialPassingCount === 'number';
const headerSummary = commercialOk
  ? totalBlocked === 0
    ? `${commercialPassingCount} of 5 commercial candidates passed both validation stages`
    : `${commercialPassingCount} of 5 commercial-OK, but ${totalBlocked} blocked by LLM verdict`
  : `${totalBlocked} query/queries blocked by validator`;
```

После Фикса 1+2 это становится тривиально: substitution и main validation согласованы → если 5 of 5 commercial OK → totalBlocked === 0 → recovery panel вообще не появляется. Заголовок показывает реальную картину когда что-то всё-таки заблокировано (после Фикса 1+2 это будет редко — только когда LLM сказала `valid:false`).

`commercialPassingCount` после Фикса 2 будет считать запросы которые прошли **обе** проверки (substitution PASS теперь учитывает `valid`). Семантика поменялась — это «прошли всю валидацию», не только commercial-only.

## Что обновляем в skill `cli-walkthrough`

Добавляю новый probe — **«согласие правил между substitution и main validation»**:

> Если substitution-блок и главная проверка применяют разные критерии — это будет приводить к ситуации «substitution думает прошло, главная блокирует». Класс бага повторялся в 1.0.4, 1.0.5, 1.0.6, 1.0.7. Для каждого validator field (`valid`, `confidence`, `search_behavior`) пройди по обоим путям и убедись что они смотрят на **тот же** field с **тем же** правилом.

И добавляю **probe «render label = настоящая причина»**:

> Для каждого пути блокировки (staticIssues / llmIssues / informationalIssues) — какая подпись отрендерится? Они **должны различаться**. Если все три пути ведут к одной подписи — будет повторение «non-commercial (retrieval-triggered)» противоречия.

## Что увидит пользователь после 1.0.8

В типичном кейсе (commercial запросы, retrieval-triggered, valid:true даже с confidence 0.6):
- substitution принимает все 5 → главная проверка тоже принимает все 5 → recovery panel вообще не показывается → пользователь видит чистый success path.

В редком кейсе когда LLM **реально** сказала `valid:false`:
- substitution не пропускает → выбирает passing spare → главная проверка тоже подтверждает.
- Если passing spare нет (все 5 invalid) → recovery panel с честной подписью «LLM rejected: <reason>» и честным заголовком «X of 5 commercial-OK, Y blocked by LLM verdict».

## Риски

1. **`valid:true` с `valid:false` рассогласование между runs (LLM стохастика)** — мы доверяем `valid` field, но LLM может выдать `valid:true` на одном run и `valid:false` на следующем для того же запроса. Митигация: validationCache persists первый вердикт; повторный run использует cached. Уже работает.

2. **Тесты на confidence threshold ломаются** — `test/validate-queries.test.js:160` и возможно другие. Все обновляются как часть Фикса 1.

3. **Какие-то downstream consumers зависят от `commercialPassingCount` == «прошли только commercial-only»** — после Фикса 2 семантика меняется на «прошли всю валидацию». Проверить: только `formatRecoveryPanel` потребляет этот field (Фикс 4 учитывает).
