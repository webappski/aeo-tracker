# 1.0.6 — Commercial-only over-generate + silent substitution

**Status**: REV 2 — addressing 9 architect remarks from REV 1 review (incl. 2 BLOCKERs: research.js:94 threshold would always throw at filter; recovery-panel header formula was wrong math)
**Target version**: `1.0.6`
**Target publish**: same day as approval

---

## Why now

Across 1.0.2 → 1.0.5, every release tried to patch the symptom: recovery panel suggests command → CLI rejects it. The user's typelessform.com dogfood on 1.0.5 caught yet another instance — option 1 with three `(validated)` keywords (from the validated pool, all three passed Fix A's both-stages check), re-ran via `--keywords`, and `voice form widget versus conversational form builder` was re-blocked at re-validation as `search_behavior: mixed`.

**Root cause** (architectural, not a bug):

The pipeline generates 4 intent buckets (commercial / vertical / comparison / problem) × 5 queries = 20 candidates, then `selectTopThree` picks ONE per bucket (REQUIRED_INTENTS = commercial+vertical+problem). Vertical / comparison / problem queries routinely fail the downstream commercial-only validator because they are not vendor-listing queries by design. So 1 of the 3 selected queries fails almost every init, and we keep building safety nets around that failure mode.

**Maintainer's directive** (saved as `memory/project_commercial_only_over_generate.md`):

> «Давай только commercial, но только мы будем их запрашивать пять, и если какой-то из commercial не проходит, мы их будем подменять. Но юзеру мы не будем показывать, пока действительно не завалились — проходят три из пяти, тогда мы не будем юзеру показывать, а просто тихонько работаем.»

Translation:
- Generate **only commercial** queries (drop vertical / comparison / problem buckets)
- Generate **5** (3 needed + 2 spares) to absorb LLM stochasticity
- Validate all 5 through both stages at init
- **Silent substitution**: if a top-3 fails, swap with a passing spare without showing anything to the user
- Recovery panel fires **only when fewer than 3 of 5 survive** — the genuine impossibility case

Closes the entire class of «recovery panel suggested command rejected» trust failures by removing the upstream cause.

## Scope

### IN

1. **`brainstorm.js`** — `INTENT_BUCKETS = ['commercial']` (was 4 buckets). Generate 5 commercial queries in a single LLM call. Output shape stays the same (`{buckets, flat}`), but `buckets.commercial` carries all 5 entries and `buckets.problem/vertical/comparison` are explicitly seeded as empty arrays in `validateBrainstormShape` for backward-compatibility (REV 2 architect catch: the loop iterates `INTENT_BUCKETS`, so without explicit seeding the other keys vanish from output).

**1a. `research.js`** — REV 2 BLOCKER: lower filter threshold at `:94` from `< 6` to `< 3`. The current threshold was sized for 20-candidate brainstorm output; with 5 it always throws. Also remove the `checkVerticalDiversity(filtered)` call at `:86` — it emits a spurious "no vertical-intent candidates" warning when only `commercial` exists in `INTENT_BUCKETS`.
2. **Prompt** — rewritten to ask for 5 commercial queries (no per-bucket distribution), in the same buyer-intent phrasing.
3. **`select.js::selectTopThree`** — pick top-3 by score from the (validated, simulated) commercial pool. Drop `REQUIRED_INTENTS`/`pickForIntent` per-bucket logic. `alternatives` = remaining ≥2 by score (typically exactly 2).
4. **New silent-substitution step in `cmdInit`** — after `selectTopThree` returns `{selected: [3], alternatives: [2]}`, run `runTwoStageValidation` on all 5. If any of top-3 fails commercial-only (`search_behavior !== RETRIEVAL`), swap with a passing alternative. Repeat until top-3 all pass OR alternates exhausted. Print NOTHING to the user — silent.
5. **Recovery panel fires ONLY when <3 of 5 pass** — the genuine impossibility. Same panel, just rarely seen. Message updated to reflect honesty: REV 2 architect catch — the original plan's formula `5 - dedupedBlockers.length` was math-wrong (dedupedBlockers counts blocked **selected** queries, ≤3, not failures-of-5). Fix: thread `passingCount` from the substitution block into `runValidationWithRecovery` → `formatRecoveryPanel`, and use it directly. Message: `Cannot auto-recover — only ${passingCount} of 5 commercial candidates passed validation; need ≥3.` When `passingCount` is not threaded (legacy paths, e.g. `--keywords` mode), fall back to the existing wording.
6. **Pool persistence** — config.candidatePool keeps the (now smaller) alternatives + any RETRIEVAL queries that became spares post-substitution.
7. **Tests** — update bucket-related assertions in `test/validator-recovery.test.js` (intent-diversity case becomes vacuous: all queries commercial). Add new `test/silent-substitute.test.js` covering: top-3 all pass → no substitution, 1 of top-3 fails → swap with first spare, 2 fail → swap both, 3 fail → recovery panel. Add new `test/brainstorm-commercial-only.test.js` with REV 2 negative-regression assertion: `flat.every(c => c.intent === 'commercial')`.
8. **Symmetric in `--queries-only` branch** of cmdInit.
9. **REV 2: Delete obsolete code path** — `lib/init/research/pool-topup.js` becomes unreachable after silent-substitution replaces 1.0.5 top-up. Delete the module + `test/pool-topup.test.js` + the `test:pool-topup` entry in `package.json`. Architect rationale: "regression guard for a code path that no longer exists is dead weight".
10. **REV 2: Drop unused exports from `select.js`** — `REQUIRED_INTENTS`, `FALLBACK_CHAIN`, `verticalDominance`. Verified no external consumers via grep.

### OUT (deferred)

- LLM verdict-consistency between runs (architecture limitation of LLM stochasticity). 1.0.6 mitigates by validating-all-5-then-substituting (over-generation absorbs variance); 1.0.7+ could persist verdicts to `validationCache` for cross-run reuse.
- Drop Phase 4 validate-category / Phase 5 simulate — they overlap with the new commercial-only-at-init validation. Big refactor; 1.0.7+.
- UX redesign of validator output (still queued from 1.0.2).
- README quickstart `--keywords` mention (queued from 1.0.3).
- CLI hygiene pass (`--json`, exit codes, `--verbose`).

### `select.js::selectTopThree` — pick-by-score, drop REQUIRED_INTENTS

**File**: `lib/init/research/select.js`

**Changes**:

```js
// drop line 23: const REQUIRED_INTENTS = ['commercial', 'vertical', 'problem'];
// drop lines 25-29: INTENT_FALLBACK_CHAIN

// selectTopThree (lines 91-150) — replace per-bucket loop with score-rank:

export function selectTopThree(candidates, options = {}) {
  const validationSkipped = options.validationSkipped === true;
  const validatedOnly = !validationSkipped;

  const sortedByScore = [...candidates]
    .filter(c => !validatedOnly || c.validation === 'ok')
    .sort((a, b) => (b.score ?? 0) - (a.score ?? 0));

  if (sortedByScore.length < 3) {
    return {
      selected: [],
      alternatives: [],
      warnings: [`only ${sortedByScore.length} validated commercial candidates — pipeline failed to generate enough`],
      infos: [],
    };
  }

  const selected = sortedByScore.slice(0, 3).map(c => ({
    intent: c.intent || 'commercial',
    candidate: c,
    fallbackUsed: null,
  }));
  const alternatives = sortedByScore.slice(3).map(c => ({
    ...c,
    unverified: c.validation !== 'ok',
  }));

  return { selected, alternatives, warnings: [], infos: [] };
}
```

**`formatSelection`** (lines 167–192): no algorithmic change. Still renders `(validated)` / `(unverified)` per current Fix A.1d logic.

### New silent-substitution step in `cmdInit`

**File**: `bin/aeo-tracker.js`

**Location**: Between current Fix A pool-validation pass (line ~1296+) and current top-up pass (line ~1350+). Replace BOTH (the top-up logic from 1.0.5 becomes obsolete when over-generation does the same job in advance).

**Sketch**:

```js
// === Replace Fix A pool-validation + 1.0.5 top-up with this single block ===
//
// 1.0.6 commercial-only + silent substitution:
//
// At this point selectResult = { selected: [3], alternatives: [2] } — all
// from a commercial-only brainstorm. We have exactly 5 candidates. Validate
// all 5 in one runTwoStageValidation call, then silently substitute any
// of the top-3 that failed with a passing alternative.

const primaryForValidation = researchProviders[0];
const allFive = [
  ...selectResult.selected.map(s => s.candidate),
  ...selectResult.alternatives,
];

if (primaryForValidation?.providerCall && allFive.length >= 3) {
  try {
    const { runTwoStageValidation, SEARCH_BEHAVIORS } =
      await import('../lib/init/research/run-validation.js');
    const validation = await runTwoStageValidation({
      queries: allFive.map(c => c.text),
      brand, domain, category: categoryDescription,
      geography: geoTags || [],
      primary: primaryForValidation,
      secondary: null,
      validationCache: [],
      commercialOnly: false,
    });
    const verdicts = validation.updatedCache || [];

    // Attach search_behavior on each candidate.
    for (const c of allFive) {
      const verdict = verdicts.find(v => v.query === c.text);
      if (verdict) {
        c.search_behavior = verdict.search_behavior;
        c.confidence = verdict.confidence;
      }
    }

    // REV 2 architect catch: if verdicts came back empty (validator returned
    // [] without throwing — e.g. throttled), treat as validation skipped
    // rather than trivially passing all 5. This prevents silent acceptance
    // of un-validated queries.
    if (verdicts.length === 0) {
      throw new Error('Validation returned no verdicts — treating as skipped');
    }

    // Silent substitution: split into passing / failing.
    const PASS = (c) =>
      c.search_behavior === SEARCH_BEHAVIORS.RETRIEVAL
      // Tolerate missing verdict for individual entries when verdicts.length>0
      // (validator handled the query but didn't classify). The (validated)-label
      // fallback in select.js matches this behaviour.
      || (verdicts.find(v => v.query === c.text) && !c.search_behavior);
    const passing = allFive.filter(PASS);
    const failing = allFive.filter(c => !PASS(c));
    const passingCount = passing.length;

    if (passingCount >= 3) {
      // Happy path — re-sort passing by score, take top-3, rest as alts.
      passing.sort((a, b) => (b.score ?? 0) - (a.score ?? 0));
      selectResult.selected = passing.slice(0, 3).map(c => ({
        intent: c.intent || 'commercial', candidate: c, fallbackUsed: null,
      }));
      selectResult.alternatives = passing.slice(3);
      // failing[] are dropped silently — user never sees them.
    }
    // ELSE: <3 passing — leave selectResult as-is. Set passingCount on the
    // result so downstream recovery panel can use the honest "X of 5" message.
    selectResult.commercialPassingCount = passingCount;
  } catch (err) {
    console.error(`${c.yellow}  Validation skipped: ${errMsg(err)}${c.reset}`);
  }
}

// Display selected — formatSelection sees the post-substitution result.
for (const line of formatSelection(selectResult)) console.log(line);
```

## Risk + rollback

### Risks

1. **Vertical / comparison / problem queries had legitimate value for some brands.** Users tracking AEO across multiple verticals may have relied on the `vertical` slot to capture niche queries. Mitigation: documented in CHANGELOG as a deliberate behaviour change; operators who need broader intent coverage can pass `--keywords` directly with a custom mix. Pre-1.0.6 configs still load; only NEW init runs use commercial-only.

2. **Brainstorm prompt regression — LLM might generate <5 valid commercial queries.** Single-bucket prompt produces less variety; LLM may repeat itself. Mitigation: existing filter.js dedup step (line 61–65) catches duplicates; if fewer than 5 unique survive, `TARGET_TOTAL_MIN = 5` threshold trips and the pipeline errors out with the existing "not enough candidates" message.

3. **`selectTopThree` change breaks downstream consumers expecting bucket-keyed selection.** Verified during reconnaissance: only `bin/aeo-tracker.js::cmdInit` consumes the output. Output shape unchanged.

4. **Silent substitution masks real LLM-generation problems.** If 3 of 5 fail every init, the user never sees why — they just get worse-than-expected queries. Mitigation: when substitution fires, log a one-line note to stderr (gated by `AEO_DEBUG=1`) so operators can opt-in to visibility.

### Rollback

If 1.0.6 produces narrower-than-expected query coverage: publish `1.0.7` reverting brainstorm.js INTENT_BUCKETS to the 4-bucket model + reverting select.js to per-required-intent picking. Silent-substitution code can stay (it's harmless on a 4-bucket pipeline; just rarely fires).
