## Why now

`cli-walkthrough` skill (newly created Step 4c) ran on `1.0.4`-as-implemented and found 2 FAIL cells in the recovery-panel 4×2 matrix:

| Cell | pool size (RETRIEVAL after Fix A filter) | clean category | option 1 emits | CLI's own gate verdict |
|---|---|---|---|---|
| **D** | 1 | no | `--keywords="q1"` (1 query) | rejected (`--keywords` requires exactly 3) |
| **F** | 2 | no | `--keywords="q1,q2"` (2 queries) | rejected |

Cell F is the bug the maintainer hit on `typelessform.com`. Cell D is the symmetric case — no test coverage today.

This is the same CLASS of trust violation 1.0.4 was supposed to remove. The fix shipped in 1.0.4 closed the contract for cells E (pool=2 + clean category) and G (pool=3 + any category) but left the unclean-category column open.

**Maintainer's directive**: fix it «правильно» — *«если что-то отвергается, мы должны ещё раз запросить, чтобы получить три. Само-достаточный механизм»*. Translation: do not suppress option 1; **top up the pool with newly-generated category-aware queries** until it has ≥3 RETRIEVAL passable queries. The tool autonomously recovers without telling the operator to do work.

Closes:
- `[bug · P0]` post-1.0.4 regression — `--keywords` suggestion with <3 items in cells D/F (just-discovered via cli-walkthrough)
- Reopens `[bug · architectural]` "no safe `--yes` path for new brands" — only partially closed in 1.0.4; cells D/F left a hole

Defers nothing new. The 4 deferred items from 1.0.4 (UX redesign, auto-substitute, placeholder hint, CLI hygiene) remain in `## 1.0.5 — next patch`.

## Scope

### IN

1. **Top-up generator** — new module `lib/init/research/pool-topup.js`. Single function `topUpPool({ needed, brand, domain, category, site, primary, validationCache, audienceTags, geoTags })` that:
   - Builds a focused prompt asking for `needed * 2` commercial RETRIEVAL-style queries for the brand/category
   - Single LLM call via `primary.providerCall`
   - Validates results through `runTwoStageValidation`
   - Returns top `needed` RETRIEVAL-passing queries (or fewer if validation rejected most)
2. **Wire into `cmdInit` pool-validation pass** — after Fix A's initial validation, count RETRIEVAL items. If `< 3`, call `topUpPool` for the gap, append RETRIEVAL passes to `selectResult.alternatives`. Max 1 top-up attempt per init (no infinite loop).
3. **Same wiring in `cmdInit`'s `--queries-only` branch** — symmetric, uses `existing.validationCache || []` for cache hits.
4. **Tighten `showOption1` fallback** in `validator-recovery.js` — IF top-up still failed to reach 3 (LLM error, all generated queries rejected, no primary provider), `showOption1 = (finalQueries.length === 3)`. This is the graceful safety net — option 1 never emits an invalid command, even if top-up itself silently no-ops.
5. **Tests**:
   - New `test/pool-topup.test.js` — unit tests for `topUpPool` with mocked `providerCall` covering: top-up generates N queries, top-up returns <N when validation rejects some, top-up returns [] on LLM error
   - Extend `test/validator-recovery-pool-filter.test.js` — add 2 cells D and F to the matrix, assert option 1 emits exactly 3 keywords OR is suppressed
   - Tighten `test/suggested-commands-resolvable.test.js` — extend the static-grep to ALSO count `--keywords="..."` items in every emitted suggestion, fail if any has !=3

### OUT (deferred to 1.0.5)

- Same as 1.0.4: UX redesign, auto-substitute, placeholder hint, CLI hygiene
- Multi-iteration top-up (we cap at 1 attempt; if 1 attempt fails to reach 3, fall back to suppress). Infinite top-up loops are dangerous; one iteration with a strong prompt should produce ≥3 valid queries in practice.
- Top-up for `--keywords` mode — that mode skips research and pool, so no top-up needed there.

## Fix design

### Module: `lib/init/research/pool-topup.js`

```js
// Top-up pool generator. Called when Fix A's initial pool validation
// produces fewer than 3 RETRIEVAL-passing alternatives, to keep the
// recovery panel's option-1 --keywords command honest (must contain
// exactly 3 queries).
//
// Generates `needed * 2` candidates in a single LLM call (over-generate so
// validation rejection doesn't leave us short), validates them through
// runTwoStageValidation, returns up to `needed` RETRIEVAL-passing results.

import { SEARCH_BEHAVIORS, runTwoStageValidation } from './run-validation.js';

const TOP_UP_OVER_FACTOR = 2;  // ask for 2× to absorb validator rejection
const TOP_UP_MAX_OUTPUT = 10;  // safety cap on LLM output

export function buildTopUpPrompt({ brand, domain, category, site, audienceTags, geoTags, needed }) {
  const audienceLine = audienceTags?.length > 0
    ? `Audience markers detected on the site: ${audienceTags.join(', ')}.`
    : '';
  const geoLine = geoTags?.length > 0
    ? `Geographic signals: ${geoTags.join(', ')}.`
    : '';
  const requestN = Math.min(needed * TOP_UP_OVER_FACTOR, TOP_UP_MAX_OUTPUT);

  return `You are an AEO (Answer Engine Optimization) keyword specialist topping up a query pool.

BRAND: ${brand}
DOMAIN: ${domain}
LANGUAGE: ${site?.lang || 'en'}
CATEGORY_DESCRIPTION: "${category || '(unknown — infer from brand+domain)'}"
${audienceLine}
${geoLine}

TASK — generate ${requestN} commercial vendor-listing queries that an AI answer
engine will RESOLVE BY LIVE WEB SEARCH (search_behavior: retrieval-triggered).

REQUIREMENTS:
  - Each query must be phrased the way real users search: "best X for Y",
    "top X 2026", "X consultants for Z" — NOT brand-comparison archetypes
    like "X vs alternatives" or "top X competitors" (those fail commercial-only
    on new brands).
  - Each query must trigger live retrieval (search-style verbs/phrasing),
    NOT parametric-only knowledge (definitional/explanatory phrasing).
  - Unbranded: never include the brand name or domain core.
  - Match the site's language.
  - One query per line, no numbering, no quotes.

Output exactly ${requestN} queries, one per line.`;
}

export function parseTopUpResponse(text) {
  if (!text || typeof text !== 'string') return [];
  return text.split('\n')
    .map(l => l.trim())
    .filter(l => l.length > 0 && !/^[\d.\-*]/.test(l))  // strip numbering/bullets
    .filter(l => l.length < 200)  // sanity
    .slice(0, TOP_UP_MAX_OUTPUT);
}

export async function topUpPool({
  needed, brand, domain, category, site,
  primary, audienceTags = [], geoTags = [], validationCache = [],
}) {
  if (!needed || needed <= 0) return [];
  if (!primary?.providerCall) return [];

  let candidates;
  try {
    const prompt = buildTopUpPrompt({ brand, domain, category, site, audienceTags, geoTags, needed });
    // POSITIONAL signature per existing convention in
    // lib/init/research/brainstorm.js:173 and validate-query-llm.js:156.
    // REV 1 architect caught: original draft used object form which would
    // pass {prompt:...} as positional arg 1 and undefined for apiKey/model.
    const { text } = await primary.providerCall(
      prompt, primary.apiKey, primary.model, { webSearch: false }
    );
    candidates = parseTopUpResponse(text || '');
  } catch {
    return [];
  }
  if (candidates.length === 0) return [];

  let validation;
  try {
    validation = await runTwoStageValidation({
      queries: candidates,
      brand, domain, category,
      geography: geoTags,
      primary,
      secondary: null,
      validationCache,
      commercialOnly: false,
    });
  } catch {
    return [];
  }

  const verdicts = validation.updatedCache || [];
  const passing = candidates
    .map(q => ({ text: q, verdict: verdicts.find(v => v.query === q) }))
    .filter(({ verdict }) => verdict && verdict.search_behavior === SEARCH_BEHAVIORS.RETRIEVAL);

  return passing.slice(0, needed).map(({ text, verdict }) => ({
    text,
    intent: 'commercial',         // top-ups are always commercial-bucket
    score: verdict.confidence ? Math.round(verdict.confidence * 100) : 70,
    unverified: false,
    search_behavior: verdict.search_behavior,
    confidence: verdict.confidence,
    topUp: true,                  // mark for traceability; persisted via persist-block update below
  }));
}
```

### Wire into `cmdInit` main flow

In `bin/aeo-tracker.js` immediately after the existing Fix A pool-validation block (after the `for (const a of poolAlts)` mutation loop), before `formatSelection`:

```js
// 1.0.4 pool-topup — if Fix A's filter left <3 RETRIEVAL alts, top up.
// Self-sufficient: tool generates the missing queries instead of asking
// the operator to retry with --category.
const retrievalCount = poolAlts.filter(a =>
  a.search_behavior === 'retrieval-triggered'
).length;
if (retrievalCount < 3 && primaryForPoolValidation?.providerCall) {
  const { topUpPool } = await import('../lib/init/research/pool-topup.js');
  const topUps = await topUpPool({
    needed: 3 - retrievalCount,
    brand, domain, category: categoryDescription,
    site, primary: primaryForPoolValidation,
    audienceTags, geoTags,
    validationCache: [],
  });
  if (topUps.length > 0) {
    // Dedup against existing selected + alternatives so a regenerated
    // duplicate query doesn't appear twice in the pool (architect REV 1 nit).
    const existingTexts = new Set([
      ...selectResult.selected.map(s => s.candidate.text),
      ...selectResult.alternatives.map(a => a.text),
    ]);
    const fresh = topUps.filter(t => !existingTexts.has(t.text));
    if (fresh.length > 0) {
      // Append to selectResult.alternatives so the existing pool-build path
      // (line 1284+) picks them up naturally.
      selectResult.alternatives = [...selectResult.alternatives, ...fresh];
    }
  }
}
```

## Risk + rollback

### Risks

1. **Top-up LLM call adds latency to every init that under-fills.** Mitigation: gated on `retrievalCount < 3` AND `primaryForPoolValidation?.providerCall` — typical happy path (3+ RETRIEVAL on first try) pays nothing. Worst case: +1 LLM call (~2-5 seconds on OpenAI). One-time at init, not on `run`.

2. **Top-up generates garbage queries.** Mitigation: every top-up goes through `runTwoStageValidation` before joining the pool. Only RETRIEVAL passes are kept. Validation throws → returns []. The safety-net `showOption1 = finalQueries.length === 3` ensures no invalid suggestion is rendered regardless.

3. **Top-up cost on rate-limited tiers.** A second `runTwoStageValidation` call (after the original Fix A pool validation) could trip a 429. Mitigation: caught by the try/catch in `topUpPool`, returns [] gracefully, recovery panel falls back to suppressed option 1.

4. **`topUp: true` flag in persisted pool** — downstream `run`/`report`/`diff` need to ignore this flag. Verified: pool is only read at init time (per architect's 1.0.4 stale-cache scenario audit). No downstream code reads pool fields, so adding a flag is safe.

### Rollback

If a critical issue surfaces post-publish:
- **Primary**: publish `1.0.5` with the top-up disabled (revert the `if (retrievalCount < 3 ...)` block; keep the safety-net `showOption1 = finalQueries.length === 3`). Restores 1.0.4-pre-top-up behaviour with the cell-D/F bug; documented as a known limitation in `1.0.5` notes.
- **Best-effort**: `npm unpublish aeo-platform@1.0.4` within 72h if allowed. Same npm policy constraints as before.
