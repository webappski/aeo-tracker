# v0.3.0 Architecture Migration — Detailed Plan

**Status:** Planning · **Target release:** v0.3.0 · **Author:** — · **Reviewers:** —
**Last updated:** 2026-04-23

## 1. Executive summary

The v0.2.2 release added error-handling coverage across all command paths (init research retry loop, run all-engines-failed panel, top-level catch). Functionally it works — 129 tests green, user-facing panels are actionable. But a senior-level code review uncovered three classes of technical debt that block confident long-term evolution:

1. **Regex-based error classification on provider messages** is fragile. Providers change wording every 3–6 months; our tests pin known strings, so classifier drift happens silently in production. This is a structural bug, not a rough edge.
2. **`process.exit(1)` scattered across ~30 call sites** means the top-level catch covers "unhandled" errors but not "handled-with-exit". The claim "user never sees a raw stack trace" is ~95%, not 100%.
3. **Three parallel error-handling mechanisms** (init retry, run cell-errors, top-level catch) evolved separately. They share a classifier but not an abstraction, which makes adding a new consumer (e.g., `run-manual`'s paste pipeline) require re-inventing the dispatch.

v0.3.0 fixes the structural debt without breaking existing users. The migration is phased across 6 checkpoints, each independently shippable as `0.3.0-beta.N`. GA goes out only when all phases are green and run manually against a real test brand for 7 days.

**What changes for users:**
- **Better error messages** — structured errors carry more context (HTTP status, provider code, retry hint) than today's regex-matched strings.
- **More reliable classification** — provider wording drift no longer silently breaks detection.
- **`AEO_DEBUG=1` semantics unchanged** — same discoverable in `--help`, same behavior.
- **No config changes required** — `.aeo-tracker.json` schema identical.
- **Two public-API additions** — `AeoProviderError`, `withProviderFallback` exported for future plugin authors. Not yet load-bearing, documented as "stable for public use from v0.3.0".

**What breaks (for library consumers, not CLI users):**
- Internal import paths for `classifyProviderError` shift from `lib/providers/classify-error.js` to `lib/providers/errors.js`. Old path re-exported as a deprecated alias for two minor versions.
- `process.exit(N)` inside commands disappears. Any caller that spawns the CLI and inspects exit codes keeps working (main still exits with same codes). Callers that `require('../lib/...')` programmatically (none known) get a cleaner API.

## 2. Problem statement — concrete bugs from v0.2.2

This section anchors the plan to real, reproducible issues. Each bug has a test-case tag we'll use in the migration.

### Bug P1 — regex classifier drift
```
# Reproduces via: change Anthropic error wording and observe silent mis-routing.
# As-of v0.2.2, classifyProviderError matches /credit.*balance/i for Anthropic billing.
# If Anthropic changes wording to "Your account lacks sufficient credits" — regex fails.
# Test passes (old string), real-world deployment silently routes as 'other'.
# Classification unit test: /credit.*balance/ matched — stays green.
# Integration test: absent — so regression invisible.
```
**Fix:** providers expose structured `{ code: 'billing', statusCode: 402, httpBody }`. Classifier looks up `code` first, regex is fallback for unclassified.

### Bug P2 — process.exit bypasses top-level catch
```
# Reproduces: trigger a missing-config error in `aeo-tracker run` (no .aeo-tracker.json).
# Current code in bin/aeo-tracker.js:1061: process.exit(1) after console.error.
# Top-level catch at bin/aeo-tracker.js:2756 NEVER fires for this path.
# User sees ad-hoc console.error text, not the unified panel.
# Some error paths show colored panel (cmdRun error class), others show raw console.error.
# Inconsistent UX across failure modes.
```
**Fix:** all command-internal exits throw `new AeoError({ exitCode, category, ... })`. One `process.exit` at bottom of main dispatcher.

### Bug P3 — three panels with overlapping rendering
```
# research-failure-panel (init, all providers failed)
# all-engines-failed-panel (run, all engines failed)
# unexpected-error-panel (top-level catch for non-command-classified errors)
# None of them dedupe — an init that fails + bubbles to top-level shows
# research-failure-panel followed by unexpected-error-panel stacked.
# User sees the same error explained twice in different formats.
```
**Fix:** single `formatAeoPanel({ error, context })` with context = 'init-research' | 'run-engines' | 'unexpected'. One code path, three presets.

### Bug P4 — retry loop is not reusable
```
# The retry walking PROVIDER_PRIORITY lives inline in bin/aeo-tracker.js:803-911.
# If we want the same behavior in `run` (auto-replay 429 once before marking cell as error), we'd copy-paste the loop.
# Two copies = two places to fix regressions.
```
**Fix:** extract to `withProviderFallback(providers, operation, onAttempt) → { result | allFailed }`. Both init and run use it.

### Bug P5 — no integration tests against real providers
```
# Unit tests: mocked or string-matching.
# Retry loop in bin/aeo-tracker.js:745-880 has zero direct test coverage.
# Integration tests: absent.
# Regression risk: high for provider wording changes.
```
**Fix:** recorded-response fixtures under `test/fixtures/provider-responses/{provider}/{scenario}.json` + zero-dep fetch mock that replays them.

### 3.2 Core contracts

#### AeoProviderError

```js
// lib/providers/errors.js
export class AeoProviderError extends Error {
  /**
   * @param {Object} opts
   * @param {string} opts.provider       'openai' | 'anthropic' | 'gemini' | 'perplexity'
   * @param {string} opts.code           'billing' | 'auth' | 'rate-limit' | 'server' | 'network' | 'malformed'
   * @param {number=} opts.statusCode    HTTP status if available (402, 401, 429, 5xx)
   * @param {string}  opts.message       Human-readable message (passed to Error)
   * @param {Object=} opts.raw           Raw provider response body for debugging
   * @param {boolean} opts.retryable     True only for billing/auth/rate-limit
   */
  constructor({ provider, code, statusCode, message, raw, retryable }) {
    super(message);
    this.name = 'AeoProviderError';
    this.provider = provider;
    this.code = code;
    this.statusCode = statusCode;
    this.raw = raw;
    this.retryable = retryable;
  }
}
```

**Provider emission example (openai.js):**
```js
// Before (v0.2.x):
if (res.status === 402) {
  throw new Error(`OpenAI: ${body.error.message}`);
}

// After (v0.3.0):
if (res.status === 402) {
  throw new AeoProviderError({
    provider: 'openai',
    code: 'billing',
    statusCode: 402,
    message: `OpenAI: ${body.error.message}`,
    raw: body,
    retryable: true,
  });
}
```

**Classifier becomes lookup-first:**
```js
// lib/providers/errors.js
export function classifyProviderError(err) {
  // Structured path — O(1) lookup
  if (err instanceof AeoProviderError) {
    return {
      retryable: err.retryable,
      category: err.code,
      reason: friendlyReason(err.code, err.provider),
      fixHint: fixHintFor(err.code),
      source: 'structured',
    };
  }
  // Regex fallback — for any provider path that still throws new Error
  // (e.g. third-party plugins, unmigrated code paths). Kept for two minor
  // versions, then removed.
  return regexFallback(err);
}
```

**Why this matters:** when OpenAI changes wording next time, our classifier keeps working because it reads `.code`. Regex fallback catches only truly-unstructured errors (third-party plugins, older code we haven't migrated).

#### withProviderFallback

```js
// lib/providers/with-fallback.js
/**
 * Walk providers in priority order until one succeeds. Retryable failures
 * (billing/auth/rate-limit) log the attempt and move to the next provider;
 * non-retryable failures (network, real bugs) throw immediately with the
 * full attempt history attached.
 *
 * @param {Array<ProviderDescriptor>} providers  In priority order
 * @param {Function} operation  async (provider) => result
 * @param {Object=} opts
 * @param {Function=} opts.onAttempt  Called per attempt: ({ provider, attempt, total }) => void
 * @param {Function=} opts.onFailure  Called on retryable failure: ({ provider, error, classified }) => void
 * @returns {Promise<{ result: T, attempts: Attempt[] }>}
 * @throws {AeoAllProvidersFailedError}  attempts is attached for the panel
 * @throws {AeoProviderError}            non-retryable — attempts attached before throw
 */
export async function withProviderFallback(providers, operation, opts = {}) { ... }
```

**Init usage:**
```js
const { result } = await withProviderFallback(researchProviders, async (primary) => {
  return await research({ brand, domain, ..., primary, validator });
}, { onAttempt: ({ provider }) => console.log(`trying ${provider.label}`) });
```

**Run usage (new — out of scope for v0.3.0 unless we want it; decision 3 below):**
```js
const cellResult = await withProviderFallback([provider], async (p) => {
  return await p.call(query, p.apiKey, p.model);
}, { retries: 1 });  // single retry on rate-limit
```

**Scope constraints:**
- Each provider's existing error branches (402, 401, 429, 5xx, malformed response) get restructured. 
- Providers keep their own wording in `.message` field.

**Risk:** medium. Provider modules haven't been touched in months. Wrong branches could re-throw or swallow. Mitigation: snapshot tests of current behavior before modifying, assert identical user-visible behavior after.

**Effort:** 4–6 hours including tests.

### Phase 2 — `withProviderFallback` abstraction

**Goal:** extract inline retry loop from `bin/aeo-tracker.js:803-911` into a reusable async generator.

**Files:**
| File | Change |
|---|---|
| `lib/providers/with-fallback.js` | **NEW** — `withProviderFallback(providers, operation, opts)` |
| `bin/aeo-tracker.js` | Replace inline loop with `withProviderFallback` call |
| `test/with-fallback.test.js` | **NEW** — 15 tests for retry semantics |

**Signature:**
```js
async function withProviderFallback(providers, operation, opts = {}) {
  const attempts = [];
  for (let i = 0; i < providers.length; i++) {
    const provider = providers[i];
    opts.onAttempt?.({ provider, attempt: i + 1, total: providers.length });
    try {
      const result = await operation(provider);
      return { result, attempts };
    } catch (err) {
      const classified = classifyProviderError(err);
      attempts.push({ provider: provider.name, label: provider.label, envVar: provider.envVar, error: err, classified });
      if (!classified.retryable) {
        // Non-retryable — attach context and rethrow
        err.attempts = attempts;
        throw err;
      }
      opts.onFailure?.({ provider, error: err, classified });
    }
  }
  throw new AeoAllProvidersFailedError({ attempts });
}
```

**Tests:**
1. Happy path — first provider succeeds → 1 attempt, result returned
2. Second provider succeeds after first billing-error → 2 attempts, result returned
3. All three fail billing → throws `AeoAllProvidersFailedError` with all attempts
4. Second provider throws non-retryable (TypeError) → throws immediately, attempts has 2 entries
5. Empty providers array → throws immediately
6. onAttempt called with correct count per iteration
7. onFailure called only for retryable failures
8. Provider descriptor shape validation (must have `.name`, `.label`)

**Risk:** low. Pure function, easy to test in isolation.

**Effort:** 2–3 hours.

**Ships as:** `0.3.0-beta.1`

### Phase 3 — AeoError + exit code consolidation

**Goal:** remove `process.exit(1)` from command bodies. Commands throw `AeoError`. Main dispatcher owns exit.

**Files:**
| File | Change |
|---|---|
| `lib/errors/aeo-error.js` | **NEW** — `AeoError` class |
| `bin/aeo-tracker.js` | Migrate ~30 `process.exit(N)` call sites to `throw new AeoError({...})` |
| `test/aeo-error.test.js` | **NEW** — 8 tests for class shape + serialization |

**Migration strategy for 30 call sites:** rather than one big rewrite, use a codemod approach:
1. Grep all `process.exit(\d)` and categorize by message pattern (validation / keys-missing / config / aborted)
2. For each category, replace with appropriate AeoError subclass (or inline AeoError with category tag)
3. Test each flow: trigger the error, assert panel rendering, assert exit code

Example migration of the most common pattern:
```js
// BEFORE:
if (!brand) {
  console.error(`${c.red}Brand is required${c.reset}`);
  closeRl();
  process.exit(1);
}

// AFTER:
if (!brand) {
  closeRl();
  throw new AeoError({
    exitCode: 1,
    category: 'validation',
    message: 'Brand is required',
    context: { hint: 'Pass --brand=X or drop --yes for interactive mode' },
  });
}
```

**Edge case — interactive `process.exit(0)` (user aborted):**
Treat as `AeoError({ exitCode: 0, category: 'aborted' })` — panel rendering for category `'aborted'` = "Aborted by user." with no error decoration.

**Tests:**
- AeoError serialization (inherits Error, captures exitCode/category/context)
- Main dispatcher catches AeoError, extracts exitCode, exits with same code
- Main dispatcher catches non-AeoError (TypeError), exits 1, renders 'other' panel
- `AEO_DEBUG=1` prints stack for AeoError too (not just top-level)

**Risk:** high. Touching 30 call sites across three commands (init/run/report). Mitigation:
1. Migrate one command at a time (init first, run second, report third)
2. Each command migration = separate commit, separate beta
3. Run full smoke test after each command (manual end-to-end pass)

**Effort:** 4–5 hours.

**Ships as:** `0.3.0-beta.2` (init migration) + `0.3.0-beta.3` (run + report migration)

**API:**
```js
export function formatAeoPanel({ error, context, useColor = true }) {
  const renderer = PANEL_PRESETS[context] || PANEL_PRESETS.unexpected;
  return renderer(error, { useColor });
}
```

**Presets:**
- `'init-research'` — receives `AeoAllProvidersFailedError`, renders top-up links + --keywords escape
- `'run-engines'` — receives aggregated error-results array, renders per-engine grouping
- `'unexpected'` — receives arbitrary error, renders classifier-based preset
- `'validation'` — receives `AeoError({ category: 'validation' })`, renders single-line red message

**Risk:** low. Pure rendering, testable in isolation. Main risk is maintaining identical output to existing three panels (user-visible regression).

**Mitigation:** snapshot-compare new output against existing panel output for 3 scenarios per context (9 golden files). Fail CI if output differs.

**Effort:** 3 hours.

**Ships as:** `0.3.0-beta.3`

### Phase 5 — Run command retry adoption

**Goal (optional — see Decision 3 below):** `run` uses `withProviderFallback` for per-engine calls with 1 retry on rate-limit.

**Why might we NOT want this:**
- Current UX: each engine fails independently, cell shows ERROR, user sees partial data. This is GOOD.
- Adding retry could mask transient issues user WANTS to see (e.g. detecting rate-limit trend).
- Retry doubles cost worst-case.

**Why we might want this:**
- Consistency with init ("the tool retries, you don't have to").
- Rate-limit retry after 30s usually succeeds — avoiding an ERROR cell that's actually a transient blip.

**Proposed middle ground — opt-in via flag:**
```bash
aeo-tracker run --retry-transient   # 1 retry per engine on 429, 30s delay
```
Default: no retry (preserves current UX). Advanced users opt in.

**Files:**
| File | Change |
|---|---|
| `bin/aeo-tracker.js` | Add `--retry-transient` flag. Wrap per-engine call in `withProviderFallback([engine], op, { retries: 1, delayMs: 30000 })`. |
| `test/run-retry.test.js` | **NEW** — 5 tests |

**Effort:** 2–3 hours.

**Ships as:** `0.3.0-beta.3` (flagged off by default) OR **deferred** to 0.4.0 if unclear ROI.

**Mock-fetch implementation sketch:**
```js
export function installMockFetch(scenarioMap) {
  const originalFetch = globalThis.fetch;
  globalThis.fetch = async (url, options) => {
    for (const [pattern, fixture] of Object.entries(scenarioMap)) {
      if (url.includes(pattern)) {
        return new Response(JSON.stringify(fixture.body), {
          status: fixture.status,
          headers: fixture.headers,
        });
      }
    }
    throw new Error(`Unmocked fetch to ${url}`);
  };
  return () => { globalThis.fetch = originalFetch; };
}
```

## 10. Decisions (resolved 2026-04-23)

All four decisions resolved per "best practice" directive from user. Rationale kept for future reference.

### Decision 1 — Semver: **0.3.0** ✅

Internal refactor, no CLI break. `1.0.0` commitment deferred to 0.4.0/0.5.0 after integration tests stabilize and we have ≥ 1 month of user feedback.

### Decision 2 — AeoProviderError surface: **public API** ✅

Exported from `lib/providers/index.js`, documented in README under a new "Extending aeo-tracker" section. Cost is negligible (4-field subclass), pay-off is plugin-author ergonomics as project grows.

### Decision 3 — Phase 5 (`run` retry): **defer to 0.4.0** ✅

Current per-engine ERROR cell is an intentional signal, not a bug. Retry would mask transient provider instability that users want to see. Phase 5 removed from the 0.3.0 scope below; doc kept for reference.

### Decision 4 — Integration test fixtures: **in-repo** ✅

`test/fixtures/provider-responses/` — ~100 KB total. Every contributor runs the same tests; no external-URL drift risk.
